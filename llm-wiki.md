# LLM Wiki

一種利用 LLM 構建個人知識庫的模式。

這是一個構思檔案，旨在供您複製並貼上到自己的 LLM 代理程式（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。其目標是傳達高層級的概念，而您的代理程式將與您協作構建具體細節。

## 核心理念

大多數人使用 LLM 處理文件的體驗就像 RAG：您上傳一組檔案，LLM 在查詢時檢索相關區塊並生成答案。這種方式雖然有效，但 LLM 每次提問都在從頭開始重新發現知識，沒有任何累積。如果您提出一個需要綜合 5 份文件細微差別的問題，LLM 每次都必須重新找尋並拼湊相關碎片，沒有建立起任何東西。NotebookLM、ChatGPT 的檔案上傳以及大多數 RAG 系統都是以此方式運作。

這裡的想法則不同。LLM 不僅在查詢時從原始文件中檢索，而是會 **逐步構建並維護一個持續成長的維基**。這是一個由 Markdown 檔案組成的結構化、互連集合，位於您與原始來源之間。當您新增來源時，LLM 不只是對其進行索引以供後續檢索，還會閱讀內容、提取關鍵資訊並將其整合到現有的維基中：更新實體頁面、修訂主題摘要、記錄新數據與舊主張矛盾之處，以及強化或質疑不斷演進的綜合結論。知識只會編譯一次並保持更新，而不是在每次查詢時重新推導。

這是關鍵的區別：**維基是一個持續存在且不斷疊加的產物。** 交叉引用已經建立，矛盾之處已被標記，綜合內容也已反映了您閱讀過的所有資訊。隨著您新增來源與提出問題，維基會變得越來越豐富。

您永遠（或很少）需要親自撰寫維基，LLM 會負責所有的編寫與維護。您負責尋找來源、探索並提出正確的問題，LLM 則處理所有的繁瑣工作：摘要、交叉引用、歸檔與記錄，這些工作才是讓知識庫長期發揮作用的關鍵。在實踐中，我會在一側開啟 LLM 代理程式，另一側開啟 Obsidian。LLM 根據我們的對話進行編輯，而我即時瀏覽結果：點擊連結、查看關係圖譜並閱讀更新後的頁面。Obsidian 是 IDE；LLM 是程式設計師；維基則是程式碼庫。

這可以應用於許多不同的場景。以下是一些範例：

* **個人**：追蹤個人目標、健康、心理、自我提升。歸檔日記、文章、播客筆記，並隨著時間建立起關於自己的結構化圖景。
* **研究**：在數週或數月內深入研究某個主題。閱讀論文、文章、報告，並逐步構建具有演進論點的全面維基。
* **閱讀書籍**：隨著進度歸檔每一章節，為角色、主題、劇情線及其關聯建立頁面。讀完後，您將擁有一個內容豐富的配套維基。想想像是 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 這樣的粉絲維基，擁有數千個涵蓋角色、地點、事件與語言的互連頁面，由志願者社群耗時多年建成。您也可以在閱讀時親自建立類似的東西，並由 LLM 處理所有的交叉引用與維護。
* **企業或團隊**：由 LLM 維護的內部維基，由 Slack 對話串、會議記錄、專案文件、客戶電話等內容提供養分。可能會有專人參與審核更新。維基能保持更新，是因為 LLM 承擔了團隊中沒人想做的維護工作。
* **競爭分析、盡職調查、旅遊規劃、課堂筆記、愛好深度研究**：任何需要隨著時間累積知識且希望資訊是有組織而非分散的場景。

## 架構

包含三個層次：

**原始來源**：您策劃的來源文件集合。文章、論文、圖像、數據檔案。這些是不可變的，LLM 從中讀取但絕不修改，這是您的事實來源。

**維基**：由 LLM 生成的 Markdown 檔案目錄。摘要、實體頁面、概念頁面、比較、概覽與綜合分析。LLM 完全擁有這一層。它會建立頁面，在有新來源時更新頁面，維護交叉引用並保持一致性。您閱讀，LLM 撰寫。

**架構定義**：一份告知 LLM 維基結構、約定以及在攝取來源、回答問題或維護維基時應遵循之工作流程的文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）。這是關鍵的配置檔案，它使 LLM 成為一個守紀律的維基維護者，而不僅僅是通用的聊天機器人。隨著您找出適合自己領域的方法，您與 LLM 將共同演進這份文件。

## 操作流程

**攝取**：您將新來源放入原始集合中，並指示 LLM 進行處理。範例流程如下：LLM 閱讀來源、與您討論關鍵重點、在維基中撰寫摘要頁面、更新索引、更新整個維基中相關的實體與概念頁面，並在記錄檔中新增一筆條目。單一來源可能會更動 10 到 15 個維基頁面。我個人偏好一次攝取一個來源並保持參與：閱讀摘要、檢查更新，並引導 LLM 強調哪些內容。但您也可以在較少監督的情況下批量攝取多個來源。您可以根據自己的風格開發工作流程，並將其記錄在架構定義中，供以後的對話使用。

**查詢**：您針對維基提出問題。LLM 搜尋相關頁面、閱讀內容並合成帶有引用來源的答案。答案的形式可以根據問題而有所不同：Markdown 頁面、比較表、投影片（Marp）、圖表（matplotlib）或畫布。重要的見解是：**好的答案可以作為新頁面歸檔回維基中。** 您要求的比較、分析或發現的關聯都是有價值的，不應消失在聊天記錄中。這樣一來，您的探索就能像攝取的來源一樣，在知識庫中不斷疊加。

**檢查**：定期要求 LLM 對維基進行健康檢查。尋找頁面之間的矛盾、已被新來源取代的陳舊主張、沒有內連連結的孤立頁面、提到但缺乏專屬頁面的重要概念、缺失的交叉引用，以及可以透過網路搜尋填補的數據空白。LLM 擅長建議可研究的新問題與可尋找的新來源，這能讓維基在成長過程中保持健康。

## 索引與記錄

兩個特殊檔案能幫助 LLM（以及您）在維基成長時進行導航。它們有不同的用途：

**index.md** 是以內容為導向的。它是維基中所有內容的目錄：列出每個頁面的連結、一行摘要，以及可選的日期或來源數量等元數據。按類別（實體、概念、來源等）組織。LLM 在每次攝取時都會更新它。回答查詢時，LLM 會先閱讀索引以找到相關頁面，然後再深入研究。這在中等規模（約 100 個來源、數百個頁面）下運作得非常好，且不需要建立基於嵌入的 RAG 基礎架構。

**log.md** 是按時間順序排列的。它是發生了什麼事以及何時發生的追加式記錄：攝取、查詢、檢查。一個實用的技巧：如果每筆條目都以一致的前綴開頭（例如 `## [2026-04-02] ingest | 文章標題`），則記錄檔可以使用簡單的 Unix 工具進行解析：`grep "^## \[" log.md | tail -5` 可讓您查看最後 5 筆條目。記錄檔提供了維基演進的時間軸，並幫助 LLM 了解最近做了哪些工作。

## 選配：命令列工具

在某些時候，您可能想要建立一些小工具來幫助 LLM 更高效地操作維基。維基頁面的搜尋引擎是最明顯的需求：在小規模下，索引檔案就足夠了，但隨著維基成長，您會需要正式的搜尋。 [qmd](https://github.com/tobi/qmd) 是一個不錯的選擇：它是一個用於 Markdown 檔案的本地搜尋引擎，具有混合 BM25 與向量搜尋以及 LLM 重新排序功能，且都在設備上運行。它同時具有命令列介面（讓 LLM 可以調用）與 MCP 伺服器（讓 LLM 可以將其作為原生工具使用）。您也可以自己建立更簡單的工具：LLM 可以根據需求協助您編寫一個簡單的搜尋腳本。

## 小技巧

* **Obsidian Web Clipper** 是一個瀏覽器擴充功能，可將網路文章轉換為 Markdown。這對於快速將來源加入原始集合非常有用。
* **將圖像下載到本地**：在 Obsidian 的「設定」→「檔案與連結」中，將「附件資料夾路徑」設定為固定目錄（例如 `raw/assets/`）。接著在「設定」→「快捷鍵」中，搜尋「Download」以找到「下載當前檔案的附件」並綁定快捷鍵（例如 Ctrl+Shift+D）。擷取文章後按下快捷鍵，所有圖像都會下載到本地磁碟。這是選配但很有用，它讓 LLM 可以直接查看並引用圖像，而不是依賴可能失效的網址。請注意，LLM 無法在一次處理中原生讀取帶有內嵌圖像的 Markdown：解決方法是讓 LLM 先閱讀文本，然後再單獨查看部分或全部引用的圖像以獲取額外背景。這雖然有點繁瑣，但運作良好。
* **Obsidian 的關係圖譜（Graph View）** 是查看維基形狀的最佳方式：哪些內容是相互關聯的、哪些頁面是中心樞紐、哪些是孤立頁面。
* **Marp** 是一種基於 Markdown 的投影片格式，Obsidian 有對應的插件。這對於直接從維基內容生成簡報非常有用。
* **Dataview** 是一個 Obsidian 插件，可以在頁面的 YAML 區塊（Frontmatter）上執行查詢。如果您的 LLM 為維基頁面添加了 YAML 區塊（標籤、日期、來源計數），Dataview 就能生成動態表格與清單。
* 維基本質上就是一個 Markdown 檔案的 Git 儲存庫。您可以免費獲得版本歷史、分支與協作功能。

## 為什麼有效

維護知識庫最乏味的部分不是閱讀或思考，而是記錄工作。更新交叉引用、保持摘要更新、記錄新數據與舊主張的矛盾，以及保持數十個頁面的一致性。人類會放棄維基，是因為維護負擔的成長速度超過了其價值。LLM 不會感到厭倦，不會忘記更新交叉引用，並且可以在一次處理中更動 15 個檔案。維基能保持維護，是因為維護成本幾乎為零。

人類的工作是策劃來源、指導分析、提出好問題並思考這一切的意義。LLM 則負責其他所有工作。

這個想法在精神上與 Vannevar Bush 的 Memex（1945 年）相關：一個個人化的、經過策劃的知識存儲，文件之間具有關聯路徑。Bush 的願景比現在的網路更接近這種模式：私密的、積極策劃的，且文件之間的連結與文件本身同樣具有價值。他無法解決的部分是誰來負責維護，而 LLM 處理了這一點。

## 備註

這份文件刻意保持抽象。它描述的是概念，而非特定的實作。確切的目錄結構、架構約定、頁面格式、工具選擇，所有這些都取決於您的領域、偏好以及您選擇的 LLM。上述提到的所有內容都是可選且模組化的：挑選有用的部分，忽略不適用的。例如：您的來源可能全是文字，因此完全不需要處理圖像。您的維基可能很小，只需要索引檔案而不需要搜尋引擎。您可能不在意投影片，只需要 Markdown 頁面。您可能想要一套完全不同的輸出格式。使用本文件的正確方式是將其分享給您的 LLM 代理程式，並共同協作實例化一個符合您需求的版本。這份文件的唯一工作是傳達這種模式，其餘部分交給您的 LLM 即可。
